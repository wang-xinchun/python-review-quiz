# 项目范围审查与知识点覆盖

## 1. 审查目的

本文件用于重新审查“Python课程智能复习与自测系统”的目标范围，确保项目难度适中：

1. 不过于复杂，避免做成大型在线教育平台。
2. 不过于简单，避免退化成普通刷题脚本。
3. 尽可能覆盖课程 PPT 中讲授的 Python 基础知识。
4. 保证最终系统能够运行、能够演示、能够写成完整课程设计报告。

## 2. 课程知识点复核

根据课程 PPT，课程主要覆盖以下内容：

1. Python 导论与学习方法。
2. 标识符、关键字、数据类型、常量、变量、运算符、表达式、类型判断和类型转换。
3. 顺序结构、选择结构、循环结构、编码规范。
4. 列表、元组、字典、集合、字符串。
5. 函数定义与调用、函数参数、匿名函数、函数嵌套与递归、变量作用域、常用系统函数、模块与包。
6. 类的定义和实例化、数据成员与方法成员、访问权限、属性、继承、多态、特殊方法和运算符重载。
7. 文件打开与关闭、文本文件读写、文件常用方法、with 上下文管理、二进制文件、文件与目录操作、文件使用示例。

由此可见，本课程更重视 Python 基础综合应用，而不是复杂 Web 后端、分布式系统或大型数据库工程。因此本项目应以 Python 基础能力展示为核心，界面框架只作为展示载体。

## 3. 总体审查结论

原定项目“Python课程智能复习与自测系统”方向合适，但需要控制范围。

推荐定位：

> 一个面向本课程复习的轻量级 Python 应用，重点完成题库管理、自测练习、错题记录、学习统计和报告导出，并通过模块化代码、类设计、文件/数据库操作体现 Python 基础知识。

不推荐定位：

> 一个复杂在线教育平台、多人登录系统、实时 AI 问答平台或完整学习管理系统。

## 4. 功能范围分级

### 4.1 必做功能

这些功能必须完成，是项目的最小闭环。

| 功能 | 说明 | 课程知识点 |
| --- | --- | --- |
| 题库初始化 | 提供一批 Python 课程题目 | 列表、字典、字符串、文件 |
| 题库浏览 | 按章节、题型查看题目 | 条件判断、循环、列表筛选 |
| 自测练习 | 选择章节后随机抽题并答题 | 随机数、循环、条件判断 |
| 自动判分 | 对比用户答案和正确答案 | 字符串处理、类型转换、表达式 |
| 错题记录 | 自动保存答错题目 | 文件/数据库写入、字典 |
| 学习统计 | 统计正确率、错题数、章节表现 | 列表、字典、聚合计算 |
| 基础界面 | 提供可演示的操作页面 | 模块调用、函数组织 |

### 4.2 应做功能

这些功能能明显提升完整度，时间允许时应完成。

| 功能 | 说明 | 课程知识点 |
| --- | --- | --- |
| 题目新增 | 在界面中新增题目 | 表单处理、数据校验、文件/数据库 |
| 关键词搜索 | 搜索题干、解析或知识点 | 字符串查找、列表筛选 |
| 错题回练 | 从错题中重新组卷 | 集合去重、列表处理 |
| 章节统计图 | 展示不同章节正确率 | 字典统计、数据可视化 |
| 学习报告导出 | 导出 Markdown 或文本报告 | 文件写入、字符串格式化 |

### 4.3 可选增强功能

这些功能可以作为亮点，但不应影响核心功能完成。

| 功能 | 说明 | 取舍建议 |
| --- | --- | --- |
| PDF 资料导入 | 从课程 PPT/PDF 提取文本 | 作为增强，不作为第一版核心 |
| AI 生成题目候选 | 用大模型辅助生成题库 | 只生成候选，必须人工检查 |
| 多题型类继承 | 不同题型使用不同类 | 可少量使用，避免过度设计 |
| 学习建议生成 | 根据错题给出复习建议 | 可用规则实现，不必接入在线 API |
| 报告截图自动整理 | 自动生成截图或摘要 | 后期再做 |

### 4.4 暂不做功能

这些功能复杂度高，和课程基础目标关系不大，建议不做。

| 功能 | 不做原因 |
| --- | --- |
| 多用户注册登录 | 增加权限、会话和安全复杂度，课程价值不高 |
| 在线部署和公网访问 | 环境配置成本高，非课程重点 |
| 实时调用大模型 API | 依赖网络和密钥，稳定性不可控 |
| 自动批量从 PDF 生成全部题目 | 容易产生错误题目，验证成本高 |
| 复杂后台管理系统 | 容易偏离 Python 基础综合应用 |
| 过度美化界面 | 报告和演示够用即可，不应占用主要时间 |

## 5. 推荐最终功能边界

建议最终系统包含 5 个主要页面或功能区：

1. 首页概览：显示题目数量、练习次数、正确率、错题数量。
2. 题库管理：浏览题目、新增题目、按章节筛选。
3. 自测练习：选择章节和题量，答题并提交。
4. 错题本：查看错题、按章节回练。
5. 学习统计与报告：显示统计图表，导出学习报告。

这样既能形成完整软件，也不会过于庞大。

## 6. 知识点覆盖设计

### 6.1 基础语法与变量

应用位置：

1. 题目数量、得分、正确率等变量计算。
2. 章节名称、题干、选项、答案等字符串变量。
3. 表单输入结果的类型转换和合法性判断。

报告写法：

> 系统在自测判分、统计计算和数据校验中使用了变量、表达式、类型判断和类型转换等 Python 基础语法。

### 6.2 顺序、选择和循环结构

应用位置：

1. 系统按“读取题目 - 展示题目 - 接收答案 - 自动判分 - 保存记录”的顺序执行。
2. 判分时使用选择结构判断答案是否正确。
3. 遍历题目、错题和练习记录时使用循环结构。

报告写法：

> 自测流程体现了典型的顺序结构、选择结构和循环结构，是课程流程控制知识的综合应用。

### 6.3 列表

应用位置：

1. 保存当前测试题目列表。
2. 保存每道题的选项。
3. 保存筛选后的题目结果。
4. 保存答题记录列表。

建议代码体现：

```python
selected_questions = random.sample(question_list, quiz_size)
```

### 6.4 元组

应用位置：

1. 用于保存固定选项，如题型集合。
2. 用于数据库查询返回的不可变记录。
3. 用于函数返回多个统计值。

建议代码体现：

```python
QUESTION_TYPES = ("single", "judge", "blank")
```

### 6.5 字典

应用位置：

1. 表示单道题目的结构化信息。
2. 按章节统计正确数和总题数。
3. 保存用户提交的答案映射。

建议代码体现：

```python
answers = {question_id: user_answer}
chapter_stats = {"第2章": {"correct": 5, "total": 8}}
```

### 6.6 集合

应用位置：

1. 去重章节名称。
2. 去重错题编号。
3. 统计已经练习过的知识点。

建议代码体现：

```python
chapters = sorted({question.chapter for question in questions})
```

### 6.7 字符串

应用位置：

1. 关键词搜索题干和解析。
2. 解析用户填空答案。
3. 生成学习报告文本。
4. 处理中文章节名和中文题目。

报告中可以强调：

> 项目包含中文路径、中文章节名和中文题目处理，能够体现字符串处理和文件编码意识。

### 6.8 函数

应用位置：

1. `load_questions()`：读取题库。
2. `filter_questions()`：筛选题目。
3. `check_answer()`：判断答案。
4. `calculate_score()`：计算分数。
5. `save_record()`：保存练习记录。
6. `generate_report()`：生成学习报告。

审查建议：

函数应保持短小清晰，每个函数只做一件事，便于报告说明和测试。

### 6.9 变量作用域

应用位置：

1. 局部变量用于单次测试过程。
2. 常量用于题型、默认题量和数据路径。
3. 避免大量全局变量，减少状态混乱。

报告写法：

> 系统将临时计算数据放在函数局部作用域中，将固定配置定义为模块常量，提高代码可维护性。

### 6.10 模块与包

应用位置：

建议目录：

```text
src/
  question_bank.py
  quiz_engine.py
  record_manager.py
  statistics_service.py
  report_exporter.py
```

对应课程知识点：

1. 自定义模块。
2. import 导入。
3. 第三方库使用。
4. 模块间职责划分。

### 6.11 面向对象

应用位置：

建议至少设计以下类：

1. `Question`：表示题目。
2. `QuizSession`：表示一次测试。
3. `StudyRecord`：表示一次练习记录。

可选设计：

1. `QuestionBank`：封装题库操作。
2. `StatisticsService`：封装统计计算。

审查建议：

不强行堆叠复杂继承。可以先用普通类或 `dataclass` 表达对象结构，再通过方法体现封装。

### 6.12 继承、多态和特殊方法

这部分可以少量体现，不宜过度复杂。

适度方案：

1. 在 `Question` 类中定义 `check_answer()` 方法。
2. 如需要支持不同题型，可定义 `SingleChoiceQuestion`、`JudgeQuestion`、`BlankQuestion`。
3. 每种题型重写 `check_answer()`，体现多态。
4. 使用 `__str__()` 方法格式化题目显示。

保守方案：

1. 使用一个 `Question` 类和 `question_type` 字段处理不同题型。
2. 报告中重点说明类、对象、属性和方法，不强行使用继承。

当前建议：

> 第一版采用保守方案，保证稳定；如果核心功能完成顺利，再加入少量题型子类作为面向对象增强。

## 7. 项目复杂度审查

### 7.1 当前设计的优点

1. 选题贴合课程，报告容易展开。
2. 功能闭环清晰，能现场演示。
3. 能自然覆盖课程大部分基础知识点。
4. 大模型协作过程有实际内容可写。
5. 后续可按时间逐步扩展，不会一开始失控。

### 7.2 当前设计的风险

1. 如果一开始处理 PDF 自动解析，可能被资料格式问题拖慢。
2. 如果加入登录、多用户、在线 AI API，复杂度会明显超出课程基础项目。
3. 如果界面做得过于花哨，容易掩盖 Python 基础能力。
4. 如果题库质量不高，系统看起来会空。

### 7.3 控制风险的方法

1. 先手工整理一批课程题目，保证题库可用。
2. PDF 导入作为增强功能，不作为第一版核心。
3. 数据库设计保持简单，避免过多表。
4. 界面采用简洁布局，重点展示流程和统计结果。
5. 每完成一个模块就记录测试结果，避免最后集中排错。

## 8. 建议的第一版验收标准

第一版完成后，应满足：

1. 程序可以正常启动。
2. 至少包含 5 个课程章节。
3. 每章至少有 5 道题。
4. 支持选择章节开始测试。
5. 能自动判分。
6. 能记录错题。
7. 能显示练习统计。
8. 能导出一份学习报告。
9. 代码至少包含 3 个自定义模块。
10. 代码至少包含 2 到 3 个类。
11. 报告中能说明对应的课程知识点。

## 9. 推荐数据规模

为避免太简单，也避免工作量过大，建议初始数据规模如下：

| 数据内容 | 推荐数量 |
| --- | --- |
| 课程章节 | 7 章左右 |
| 每章题目 | 5 到 8 道 |
| 总题目数 | 40 到 60 道 |
| 题型 | 单选题、判断题、填空题 |
| 测试题量 | 每次 5 到 10 道 |

这个规模足够演示筛选、抽题、统计和错题功能，也不会让题库整理成为主要负担。

## 10. 报告重点建议

最终报告中应突出以下内容：

1. 本系统如何服务 Python 课程复习。
2. 每个模块分别对应哪些 Python 基础知识。
3. 数据结构如何设计，为什么这样设计。
4. 函数和类如何划分，体现模块化和面向对象思想。
5. 文件/数据库如何保存题库、错题和学习记录。
6. 大模型 Agent 如何参与规划、开发、测试和报告整理。
7. 大模型输出如何经过人工审查和运行验证。

## 11. 最终审查结论

本项目目标合理，但必须坚持“轻量完整”的原则：

1. 以题库、自测、错题、统计、报告导出为主线。
2. 以 Python 基础知识覆盖为技术展示重点。
3. 以大模型协作记录为课程特色亮点。
4. 不做复杂登录、在线部署和实时 AI API。
5. 第一版先跑通核心闭环，再考虑 PDF 导入和 AI 题目生成等增强功能。

按照以上范围推进，项目既不会过于简单，也不会复杂到难以收尾，适合作为 Python 课程期末大作业。
